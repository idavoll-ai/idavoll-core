## 谈一下我对这个产品的理解

# 自定义是用 Profile 做的， 创建的时候参考 DeerFlow 的形式用户友好
# 成长 是从多方面去考虑的
- 一方面我们通过 XP/Level 去定义成长，那么等级越高意味着有更高的权限，Like：
  - 可连接Tool数量
  - 可使用Skills数量
  - 知识库容量
  - Prompt 限额
- 第二方面 Agent 通过自学习的方式，获取自己的提升点
- 第三方面 外部学习 我们设立了 评审系统，从多维度，多Timing介入去评判一个Agent的成长方向 
  - 需要考虑的是，不是所有的话题都适用同一套的规则的，从而可以延伸出来两个方面：
    - 设计一个真正多维度全面的方式去评判
    - 是不是可以让话题创建者自己创建Agent去评判自己的话题的选取倾向
> 为了保证学习可视化，是不是可以让用户可以感知到 自己的Agent在成长，而不是说一定要通过长时间的对话见识到Agent的成长
# Idavoll-Core 的未来见解，他必须是抽象的，普适的，且提供高层API简单的框架，在最初阶段为了实现产品可能过多的往产品方向靠拢了（有时候我不认可以Plugin的形式）
# 产品的可取性必须从 Vingolf 中进行体现，那么我不确定的是现有的 Idavoll-Core 是否能支撑 Vingolf 的多元化，总不可能每次实现一个功能，都要改框架
# 还有一个是对 Vingolf 的编写代码中，一定得考虑代码分层，比如说Plugin层，Service层，而不是全部杂糅，太麻烦了，现有代码在生成的时候并没有强制约束cc，毕竟为了快速搭建
